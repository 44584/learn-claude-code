# s02: Tool Use — 项目介绍

## 一句话总结

这是一个教学项目，展示如何让 AI Agent 从**只有 1 个 bash 工具**扩展到**拥有 5 个专用工具**，并实现优雅的**工具分发架构**。

## 核心架构：工具分发（Tool Dispatch）

项目引入了一个 **`TOOL_HANDLERS` 查表分发机制**，加一个新工具只需两步：

1. **定义工具**：在 `TOOLS` 数组里加一条 JSON schema 描述
2. **注册处理函数**：在 `TOOL_HANDLERS` 字典里加一行映射

Agent 主循环（`while True`）完全不变——体现了 **"加一个工具，只加一行"** 的设计哲学。

## 5 个工具一览

| 工具 | 用途 | 函数 |
|------|------|------|
| `bash` | 执行 shell 命令 | `run_bash()` |
| `read_file` | 读取文件内容 | `run_read(path, limit)` |
| `write_file` | 写入文件内容 | `run_write(path, content)` |
| `edit_file` | 替换文件中的指定文本 | `run_edit(path, old_text, new_text)` |
| `glob` | 按模式查找文件 | `run_glob(pattern)` |

## 为什么从 1 个工具扩到 5 个？

s01 的 Agent 只有 bash，读文件要拼 `cat`，写文件要拼 `echo`，改文件要拼 `sed`。模型明明想的是"读这个文件"，却要翻译成 shell 命令——浪费 token，还容易拼错。

有了专用工具，模型可以直接调用 `read_file`、`write_file`、`edit_file`，表达更自然，操作更精准。

## 多工具调用

模型可以一次返回多个 `tool_use`（例如"读 a.py 和 b.py，再列出所有 .py 文件"），教学版按返回顺序逐个执行。

## 路径安全

`read_file`、`write_file`、`edit_file` 三个文件工具受 `safe_path` 校验保护，防止路径穿越攻击。

## 与 s01 的对比

| 组件 | 之前 (s01) | 之后 (s02) |
|------|-----------|-----------|
| 工具数量 | 1 (bash) | 5 (+read, write, edit, glob) |
| 工具执行 | 硬编码 `run_bash()` | TOOL_HANDLERS 查表分发 |
| 路径安全 | 无 | safe_path 校验 |
| 循环 | `while True` + `stop_reason` | 完全一致 |
